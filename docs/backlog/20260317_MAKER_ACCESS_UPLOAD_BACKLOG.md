## 백로그 항목

### 1. 메이커를 웹으로도 안정적으로 접속 가능하게 정리

#### 현재 기준 현황

- 메이커는 별도 로컬 앱이 아니라 같은 Next.js 웹 앱 안의 라우트로 이미 존재한다.
- 라이브러리에서 [`/maker/new`](/Users/youuchul/Documents/github/00_portfolio/Murder-Mystery_Generator/src/app/library/page.tsx) 로 새 게임 생성이 연결되어 있다.
- 기존 게임 편집도 [`/maker/[gameId]/edit`](/Users/youuchul/Documents/github/00_portfolio/Murder-Mystery_Generator/src/app/maker/%5BgameId%5D/edit/page.tsx) 라우트로 바로 열린다.
- 즉 서버를 LAN 또는 터널로 열면 플레이어 화면뿐 아니라 메이커 화면도 같은 방식으로 외부 접속 가능하다.

#### 남아 있는 문제

- 현재는 메이커 접속 URL이 사용자 관점에서 정리되어 있지 않다.
- GM 보드에는 플레이어 참가 링크 중심으로만 안내가 있고, 메이커 접속 흐름은 별도 설명이 없다.
- 메이커 라우트에 인증/권한 제어가 없어 URL을 아는 사람이 바로 편집 화면에 들어올 수 있다.
- 외부 공개 시 “플레이용 공개 URL”과 “제작용 관리 URL”의 구분이 없다.
- 장기적으로는 `플레이는 전체 라이브러리`, `편집은 자기 게임만` 정책으로 가야 한다.

#### 후속 작업 제안

- 메이커 접속 방식 문서화
  - 로컬: `http://<LAN-IP>:<PORT>/library`
  - 터널: `<tunnel-url>/library`
- 메이커 진입점을 `/library` 기준으로 통일하고 안내 문구 정리
- 필요하면 GM 화면 또는 라이브러리에 “제작 URL 복사” 액션 추가
- 최소한의 접근 제어 검토
  - 간단한 비밀번호 게이트
  - 관리자 토큰
  - 공개 플레이 URL과 분리된 메이커 접근 경로

#### 우선순위

- 중간
- 외부에서 같이 제작하거나 다른 기기에서 수정할 일이 많아지면 우선순위 상승

### 2. 이미지 입력을 URL 중심에서 업로드 중심으로 확장

#### 초기 분석 당시 확인 상태

- 현재 업로드 API는 [`src/app/api/games/[gameId]/assets/route.ts`](/Users/youuchul/Documents/github/00_portfolio/Murder-Mystery_Generator/src/app/api/games/%5BgameId%5D/assets/route.ts) 에 있고, `covers` 와 `locations` scope만 지원한다.
- 실제 업로드 UI도 현재는 표지와 장소 이미지 정도만 붙어 있다.
- 아래 항목들은 여전히 URL 직접 입력 비중이 높다.
  - 대표 지도
  - 피해자 이미지
  - NPC 이미지
  - 단서 이미지
  - 플레이어 카드 이미지
  - 라운드별 대표 이미지
  - 엔딩 분기 영상/배경음악은 여전히 URL-only

#### 남아 있는 문제

- 영상/배경음악처럼 아직 URL-only 인 필드가 남아 있다.
- 라운드 대표 이미지처럼 이미지 scope는 남아 있는데 UI가 아직 없는 영역이 있다.
- 메이커 자산의 운영 정책과 보존 방식은 여전히 로컬 파일 저장 전제다.

#### 후속 작업 제안

- 에셋 업로드 scope 확장
  - `story`
  - `clues`
  - `players`
  - `rounds`
- URL 입력과 업로드를 병행 지원하되, 기본 CTA는 업로드로 전환
- 공통 업로드 컴포넌트화
  - 업로드 버튼
  - 미리보기
  - 교체
  - 제거
- 이미지 필드 우선 적용 순서
  1. 대표 지도
  2. 피해자 / NPC
  3. 단서 이미지
  4. 플레이어 카드 이미지
  5. 라운드 대표 이미지

#### 우선순위

- 중간
- 핵심 이미지 필드는 대부분 정리됐고, 남은 것은 라운드/미디어와 운영 정책 영역이다.
