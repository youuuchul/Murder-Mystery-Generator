# 2026-03-19 라이브러리 / 메이커 접근 분리 백로그

## 목적

현재는 서버를 외부에 열면 `/join` 뿐 아니라 `/library`, `/maker/*` 도 같이 열리는 구조다.
단기적으로는 외부 메이커 접속을 허용하고 싶고, 장기적으로는 이 프로젝트를 플랫폼처럼 운영할 계획이므로
`플레이는 전체 라이브러리`, `편집은 자기 게임만` 이라는 정책을 먼저 문서로 고정한다.

## 목표 제품 모델

- 누구나 라이브러리에서 공개된 게임을 보고 플레이 세션을 시작할 수 있다.
- 사용자는 자기 게임만 생성/수정/삭제할 수 있다.
- 나중에는 협업자 초대나 공동 편집을 붙일 수 있지만, 1차 목표는 `소유자 단독 편집` 이다.
- 플레이용 공개 동선과 제작용 관리 동선은 명확히 분리한다.

## 현재 상태

- 기술적으로는 같은 Next.js 앱 안에 `library`, `maker`, `join`, `play` 라우트가 모두 있다.
- 서버를 LAN 또는 터널로 열면 메이커 화면도 외부에서 직접 접근 가능하다.
- 2026-03-19 기준으로 `/library`, `/maker/*`, `/api/games/*`, `/api/maker-assistant` 에는 임시 메이커 비밀번호 게이트를 걸 수 있다.
- 이 게이트는 `게임별 비밀번호` 가 아니라 `메이커 영역 전체 공통 비밀번호` 다.
- 즉 비밀번호를 아는 사용자는 현재 모든 게임을 열고 편집할 수 있다.
- 하지만 사용자 인증, 게임 소유권, 수정 권한 검증은 아직 없다.
- 즉 현재 문제는 “외부에서 못 들어간다” 보다는 “들어가도 누가 무엇을 수정할 수 있는지 세밀하게 통제할 장치가 없다” 쪽이다.

## 오해 방지 메모

- 현재 구현된 `MAKER_ACCESS_PASSWORD` 는 테스트용 임시 출입문이다.
- 이건 “메이커 전체를 잠그는 문”이지, “게임마다 다른 열쇠”가 아니다.
- 따라서 장기 목표인 `편집은 자기 게임만` 정책을 해결한 것이 아니다.
- 이 기능을 설명할 때는 반드시 `메이커 전체 공통 비밀번호` 라고 표현한다.

## 고정할 정책

### 1. 라이브러리 노출 정책

- 라이브러리에는 `플레이 가능한 공개 게임` 이 노출된다.
- 초안 중인 게임, 개인 메모성 게임, 미공개 게임은 공개 라이브러리에서 숨긴다.
- 향후 상태 예시
  - `draft`
  - `private`
  - `public`

### 2. 편집 권한 정책

- 게임 생성자는 해당 게임의 기본 소유자다.
- 소유자만 편집, 삭제, 공개 상태 변경을 할 수 있다.
- 협업 기능은 후속 단계로 미룬다.

### 3. 플레이 권한 정책

- 공개 라이브러리에 있는 게임은 누구나 세션 시작 가능
- 세션 시작 후 실제 참가자는 기존처럼 `/join` 코드로 입장
- 편집 권한과 플레이 권한은 분리한다.

### 4. URL 정책

- 공개 플레이 동선
  - `/library`
  - `/join`
  - `/play/*`
- 제작/관리 동선
  - `/maker/new`
  - `/maker/[gameId]/edit`
  - 향후 `내 게임` 중심 관리 화면

## 핵심 백로그

### 1. 사용자 식별 체계

- 최소한의 계정 또는 관리자 토큰이 필요하다.
- 게임 소유권을 연결할 `userId` 가 없으면 “내 게임만 편집” 정책을 구현할 수 없다.
- 우선순위: 높음

### 2. 게임 소유권 필드

- `game.json` 에 소유자 정보와 공개 상태가 필요하다.
- 예시
  - `ownerId`
  - `visibility`
  - `publishedAt`
- 우선순위: 높음

### 3. 편집 API 권한 검증

- `/api/games/*` 의 생성/수정/삭제, 자산 업로드, 메이커 편집 진입에 권한 체크가 필요하다.
- UI 숨김만으로는 부족하고 서버에서 막아야 한다.
- 우선순위: 높음

### 4. 라이브러리 화면 분리

- 장기적으로는 `공개 라이브러리` 와 `내 게임 관리` 가 분리되는 편이 자연스럽다.
- 최소안
  - 공개 라이브러리 탭
  - 내 게임 탭
- 우선순위: 중간

### 5. 공개 상태 변경 UX

- 게임을 언제 공개 라이브러리에 올릴지 명시적으로 제어해야 한다.
- 초안 게임이 실수로 공개되지 않게 해야 한다.
- 우선순위: 중간

### 6. 협업자 모델

- 나중에는 공동 제작을 위해 `editor` 권한 공유가 필요할 수 있다.
- 하지만 1차 범위에서는 제외하고, 소유자 단독 편집을 먼저 닫는다.
- 우선순위: 낮음

## 추천 진행 순서

1. 사용자 식별 + 게임 소유권 필드
2. `/api/games/*` 권한 체크
3. 공개 라이브러리 vs 내 게임 UI 분리
4. 공개 상태 전환 UX
5. 협업자 초대

## 메모

- 단기 운영상으로는 이미 구현된 “메이커 비밀번호 게이트” 만으로도 외부 공동 제작 테스트는 바로 가능하다.
- 다만 이 게이트는 게임별 권한 기능이 아니라, 메이커 전체에 하나의 공통 비밀번호를 거는 수준이다.
- 하지만 플랫폼 방향을 생각하면 결국 `계정 + 소유권 + 공개 상태` 로 가야 한다.
- 따라서 임시 게이트는 빠른 운영 해법이고, 장기 해법은 이 문서의 정책 모델이다.
