# 2026-03-19 라이브러리 / 메이커 접근 분리 구현 계획

## 목표

이 문서는 아래 제품 규칙을 실제 코드와 데이터 구조로 옮기기 위한 구현 계획이다.

- 플레이는 공개 라이브러리의 게임을 기준으로 한다.
- 편집은 자기 게임만 가능해야 한다.
- 현재의 단일 앱 구조는 유지하되, 읽기/쓰기 권한을 분리한다.

## 구현 단계

### Phase 0. 임시 운영용 메이커 게이트

목표:
- 계정 시스템 없이도 외부 메이커 접속을 안전하게 제한

방법:
- `/library`, `/maker/*`, `/api/games/*` 에 공통 비밀번호 또는 관리자 토큰 체크 추가
- `/join`, 공개 플레이 세션 API는 그대로 유지

장점:
- 구현 범위가 작다.
- 다른 메이커가 외부에서 들어와 작업하는 운영 실험을 바로 할 수 있다.

한계:
- 게임별 비밀번호가 아니다.
- 메이커 영역 전체에 공통 비밀번호 하나를 거는 방식이다.
- “누가 누구 게임을 수정할 수 있는지” 까지는 구분 못 한다.
- 비밀번호를 아는 사용자는 현재 모든 게임을 편집할 수 있다.
- 플랫폼 최종 구조로는 부족하다.

구현 상태:
- 2026-03-19 기준 1차 구현 완료
- 보호 대상
  - `/library`
  - `/maker/*`
  - `/api/games/*`
  - `/api/maker-assistant`
- 예외
  - `/join`
  - `/play/*`
  - 업로드 자산 GET
- 설정 방식
  - `MAKER_ACCESS_PASSWORD`
  - 비어 있으면 게이트 비활성화

표현 규칙:
- 이 기능을 설명할 때는 `메이커 전체 공통 비밀번호` 라고 쓴다.
- `게임별 비밀번호`, `내 게임 보호` 같은 표현은 사용하지 않는다.

## Phase 1. 사용자 식별 도입

목표:
- 게임 소유권을 연결할 최소 식별 체계 확보

필요 데이터:

```ts
interface AppUser {
  id: string;
  displayName: string;
}
```

현실적인 최소안:
- 자체 계정 없이 세션/토큰 기반 관리자 로그인
- 또는 외부 auth 도입 전까지 단일 사용자 생성 흐름

이 단계 산출물:
- `userId` 를 읽을 수 있는 auth helper
- 메이커 라우트에서 로그인 여부 확인

## Phase 2. 게임 소유권 / 공개 상태 도입

목표:
- 게임별 편집 권한과 공개 라이브러리 노출 정책 정의

`GamePackage` 확장 후보:

```ts
interface GameAccessMeta {
  ownerId: string;
  visibility: "draft" | "private" | "public";
  publishedAt?: string;
}
```

적용 규칙:
- `ownerId === currentUser.id` 인 경우에만 편집 가능
- `visibility === "public"` 인 게임만 공개 라이브러리에 노출
- `draft/private` 는 소유자만 볼 수 있는 관리 목록에 남김

## Phase 3. API 권한 분리

목표:
- UI가 아니라 서버에서 실제 권한을 보장

대상:
- `POST /api/games`
- `PATCH /api/games/[gameId]`
- `DELETE /api/games/[gameId]`
- `POST /api/games/[gameId]/assets`

권한 규칙:
- 생성: 로그인 사용자만
- 수정/삭제/업로드: 소유자만
- 읽기:
  - 소유자는 자기 게임 전체 JSON 열람 가능
  - 공개 라이브러리는 공개용 요약 정보만 반환

## Phase 4. 라이브러리 UI 분리

목표:
- 플레이용 공개 라이브러리와 내 작업 관리 화면을 혼동하지 않게 분리

추천 UI:

- `/library`
  - 공개 게임 목록
  - 플레이 시작 중심
- `/library/manage` 또는 `/my-games`
  - 내가 만든 게임 목록
  - 새 게임 생성
  - 편집 / 공개 상태 변경

최소안:
- 한 화면 안에서 `공개 게임` / `내 게임` 탭 분리

## Phase 5. 공개 상태 전환 UX

목표:
- 초안 게임이 실수로 공개되지 않게 제어

필요 요소:
- `공개하기`
- `비공개로 전환`
- 공개 전 체크리스트
  - 제목/소개글
  - 최소 플레이어 수
  - 엔딩 여부
  - 기본 스크립트 여부

## 라우트 / API 영향 범위

### 메이커 라우트

- `/maker/new`
- `/maker/[gameId]/edit`

필요 변경:
- 로그인 확인
- 소유자 확인
- 권한 없으면 리다이렉트 또는 403 화면

### 라이브러리

- 공개 라이브러리 조회 API
- 내 게임 조회 API
- 카드 액션 분기
  - 플레이 시작
  - 편집
  - 공개 상태 변경

### 세션 시작 정책

- 공개 게임은 누구나 세션 시작 가능
- 비공개/초안 게임은 소유자만 세션 시작 가능하게 할지 결정 필요

추천:
- `public`: 누구나 세션 시작 가능
- `private/draft`: 소유자만 세션 시작 가능

## 데이터 마이그레이션 메모

기존 로컬 게임 JSON에는 `ownerId`, `visibility` 가 없다.

대응안:
- 기존 게임은 임시로 현재 관리자 계정 소유로 이관
- 기본 visibility 는 `private`
- 공개 전환은 사용자가 직접 수행

## 리스크

### 1. 로컬 JSON 저장소 한계

- 여러 사용자가 동시에 수정할 때 충돌 가능성이 있다.
- 플랫폼 운영 규모가 커지면 DB 전환이 필요해질 수 있다.

### 2. 공개 라이브러리와 스포일러 분리

- 플레이용 라이브러리 응답에는 민감 필드가 섞이지 않도록 sanitizing 이 중요하다.

### 3. 세션 시작 권한

- “누구나 플레이 가능” 을 어디까지 허용할지 운영 정책이 필요하다.
- 공개 라이브러리 노출과 세션 시작 권한을 같은 규칙으로 볼지 분리할지 결정해야 한다.

## 추천 실행 순서

1. 임시 메이커 비밀번호 게이트
2. 사용자 식별 도입
3. `ownerId`, `visibility` 필드 추가
4. 게임 CRUD / 업로드 권한 체크
5. 공개 라이브러리 / 내 게임 UI 분리
6. 공개 상태 전환 UX
